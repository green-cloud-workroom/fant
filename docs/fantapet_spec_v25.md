## 1. 결정 9 정정 (휴일 마스터)

spec_v23 §4 "결정 9 — 휴일 마스터" 보강. 코드 진입 시점(Phase 2)에 이 절을 단일 소스로 참조.

### 1-1. 마스터 소유권 — 옵션 A 확정

```
생산관리 앱이 모든 휴일/특수일 마스터를 단독 보유.
재고관리 앱은 조회 전용.

- 한국 법정공휴일: 생산관리 앱 자동 등록 (5년치 코드 내장)
- 회사 휴무일: 생산관리 앱에서 직접 등록
- 둘 다 같은 `holidays` 컬렉션에 저장
- 재고관리 앱은 `holidays` 컬렉션 read-only 접근
```

⚠ 재고관리 앱 spec §2.4 정정 필요:
- 현행 표현 "회사 정기 휴무일 = 재고관리 앱 직접 등록" → "생산관리 앱 직접 등록 (조회만)"으로 정정
- 재고관리 앱 화면 5-3에서 "+ 회사 휴무일 등록" 버튼 제거
- 회사 휴무일 행도 출처: "생산관리 앱" + 작업 칸 "-" (조회 전용)
- 정정 시점: 재고관리 앱 spec 다음 갱신 시. spec_v25에 정정 필요 항목으로 박힘.

### 1-2. 회사 휴무일 등록 모달 필드 (생산관리 앱)

진입 경로: 설정 → 휴일/특수일 → "+ 회사 휴무일 등록"

권한: admin/office (spec_v23 §10 권한 매트릭스 따름)

#### 필드 구성

| 필드 | 타입 | 디폴트 | 비고 |
|---|---|---|---|
| 날짜 | 단일 / 연속 범위 | 오늘 | 시작일/종료일 |
| 휴일명 | string | (없음) | 필수 |
| 메모 | string | (없음) | 선택 |
| 생산 영향 | checkbox | OFF | ☑ "이날 생산하지 않음" |
| 배송 영향 | checkbox | ON | ☑ "이날 배송하지 않음" |
| 전날 배송 | checkbox | **ON** | ☑ "휴일 전날 배송 가능" |
| 반복 | select | "한 번만" | 한 번만 / 매년 반복 |

#### 디폴트 동작 — 회사 휴무일

회사 휴무일은 **택배사가 정상 운영**한다고 가정:
- 생산은 안 하는 게 일반 (생산 영향 OFF는 운영자가 직접 ON으로)
- 배송은 안 함 (배송 영향 ON)
- **전날 배송 가능 — 택배사가 휴일에도 일하므로 전날 보내도 됨** (체크박스 ON)

운영자가 모달에서 모두 변경 가능.

#### Holiday 컬렉션 필드 매핑

```js
{
  holidayId: string,
  date: date,                              // 단일 날짜 or 범위 시 일별 분할
  holidayType: 'internalOff',              // 회사 휴무일
  title: string,                           // 휴일명
  description: string,                     // 메모
  affectsProduction: boolean,              // 생산 영향 체크박스
  affectsShipping: boolean,                // 배송 영향 체크박스
  shippingClosedFromEnabled: boolean,      // ⚠ NOT(전날 배송 가능)
                                           // = !(체크박스 값)
                                           // 체크박스 ON  → shippingClosedFromEnabled: false
                                           // 체크박스 OFF → shippingClosedFromEnabled: true
  isAutoGenerated: false,
  recurrenceRule: json | null,             // "매년 반복" 선택 시 채움
  createdBy: string,
  updatedAt: timestamp,
}
```

⚠ **`shippingClosedFromEnabled` 필드명 = 부정 표현**. UI 체크박스 라벨 (긍정 표현 "휴일 전날 배송 가능")과 부울값이 반대임. 코드에서 매핑 시 주의.

### 1-3. 한국 공휴일 자동 등록 디폴트값 정정

spec_v23 §4 A 기존 디폴트 그대로 + UI 표현 정정:

```js
{
  holidayType: 'publicHoliday',
  affectsProduction: true,                // 생산 안 함
  affectsShipping: true,                  // 배송 안 함
  shippingClosedFromEnabled: true,        // 전날 배송 불가 (택배사 휴무)
  isAutoGenerated: true,
}
```

UI 표현 ("휴일 전날 배송 가능" 체크박스 기준):
- 공휴일 디폴트: ☐ "휴일 전날 배송 가능" **OFF** (택배사 휴무라 전날도 못 보냄)
- 회사 휴무일 디폴트: ☑ "휴일 전날 배송 가능" **ON** (택배사 정상 운영)

운영자가 자동 등록된 공휴일도 UI에서 변경 가능 (spec_v23 §4 E "운영자 변경 가능 사항" 유지).

### 1-4. 영업일 계산 룰

배송 영업일 = (날짜) AND (영업일 룰):

```
영업일 = NOT (
  토요일 OR
  일요일 OR
  affectsShipping === true OR
  (전날이 holiday이고 prevDay.shippingClosedFromEnabled === true)
)
```

= 토/일/배송 불가일/전날이 "전날 배송 불가" 휴일이면 영업일 아님.

### 1-5. UI 변경 가능 사항 (이력 저장)

spec_v23 §4 E 유지. 변경 시 `activityLogs` 발행:
- 변경 필드: `affectsProduction` / `affectsShipping` / `shippingClosedFromEnabled` / 휴일명 / 메모
- 변경 전/후 값 저장
- 변경자, 일시 기록
- `action: 'holiday'`, `subAction: 'edit'`

신규 등록 시: `subAction: 'create'`. 삭제 시 (회사 휴무일만, 자동 등록 공휴일은 삭제 불가): `subAction: 'delete'` + 소프트 삭제 (`status: 'deleted'`, hard delete 안 함).

### 1-6. wipe 동작 명시

spec_v23 §6 wipe whitelist는 `holidays` 포함. 정책 보강:

- 운영자 등록 회사 휴무일 (`isAutoGenerated: false`): wipe 대상
- 자동 등록 한국 공휴일 (`isAutoGenerated: true`): **wipe 대상 제외**

wipe 스크립트 조건: `where('isAutoGenerated', '!=', true)`.

이유: 한국 공휴일은 코드 내장 데이터로 자동 재생성되긴 하지만, 운영자가 자동 등록값을 수정했을 경우 (1-5 변경) wipe로 그 수정 이력이 사라지는 게 부자연스러움. 자동 등록 데이터는 보존하고 운영자 등록 데이터만 wipe.

### 1-7. 진행 시점

- **Phase 2** (재고관리 앱 묶음 진입 직전 필수)
- 묶음 9 #22 (캘린더 휴일 표시)와 같이 작업
- 작업 예상: spec_v23 §4 F 그대로 — 16~21h (E #22 + 휴일 마스터 통합)

## 2. 운영 발견사항 #2 보강 — productions 소프트 삭제

### 정정 대상

spec_v24 §2-4 "생산 카드 삭제 시 영양제 처리"가 hard delete 가정으로 작성됨. 실제 구현은 소프트 삭제. 동작은 정상이므로 spec만 정정.

### v25 정정

```
### 2-4. 생산 카드 삭제 시 영양제 처리 (v25 정정)

생산 카드 삭제 = 소프트 삭제 (`status: 'deleted'` 갱신, doc 자체는 보존).
영양제 차감했던 1봉 환불.

처리 흐름:
1. `productions/{id}` 트랜잭션 읽기
2. `status: 'deleted'` 갱신
3. `relatedProductionId` 매칭 `supplementLogs` net 합 계산
   - sum > 0 (환불이 차감보다 많은 비정상 상태) → 삭제 차단
4. `supplementStock.currentQty += 1`
5. `supplementLogs` 발행 (`type: 'autoDeduct'`, `qty: +1`, `reason: 'production deleted'`, `relatedProductionId` 카드 ID)

⚠ hard delete 아님. doc는 status 갱신으로만 처리.

내일생산불러오기 전/후 무관 (원육/봉투와 다름). 카드 삭제 = 무조건 환불.
```

### 통계 영향

- `productions.status === 'active'`만 통계 포함 (기존 동작 유지, codebase.md 탭1 § 참조)
- 소프트 삭제된 카드는 통계 자동 제외

## 3. 운영 발견사항 #3 보강 — activityLogs delete 금지

### 신설 절 — activityLogs 정책

Firestore rules상 `activityLogs` 컬렉션은 delete 금지. 단위 11 자동 발행 로그 삭제 시뮬레이션 시 발견.

```
### activityLogs 컬렉션 정책 (v25 신설)

- create: 허용 (모든 액션에서 발행 가능)
- read: 허용 (사무/생산 로그 패널 조회)
- update: acknowledged 처리만 허용
- delete: **금지** (Firestore rules)

#### 누적 방지

자동 발행 로그 (`isAutoGenerated: true`)는 결정적 doc id 패턴으로 덮어쓰기.
같은 날짜 + 같은 dedupKey에 대해 같은 doc id가 생성되므로 중복 발행되어도 1건만 존재.

doc id 패턴: `auto_{action}_{subAction}_{date}_{dedupKey}`
예: `auto_minStock_alert_2026-05-14_supplementStock_VNGOvGiefHOXm8ldZkSP_20`

#### 장기 archival 정책 (장기 검토 항목)

- 운영 6개월~1년 후 acknowledged 로그가 대량 누적될 가능성
- 현재 삭제 불가하므로 누적 그대로 보존
- 장기 검토 시점에 다음 옵션 결정:
  (a) Firestore rules 완화하여 일정 기간 후 자동 delete
  (b) `archivedLogs` 컬렉션 신설 + 마이그레이션
  (c) acknowledged + N개월 경과 로그 frontend 필터링만 (DB 보존)
- 현재 결정: 운영 1년 시점에 누적 상황 보고 결정. spec_v26 또는 별도 정책 문서.

#### 검증 시 주의

검증 중 발행된 테스트 로그가 삭제 불가. 정리 방법:
- acknowledged 처리만 가능 (필터에서 사라짐)
- doc 자체는 영구 보존
- 시드 입력 전 wipe 시 `activityLogs`도 whitelist에 포함되어 전체 삭제됨 (wipe 스크립트 권한)
```

## 4. 운영 발견사항 #6 보강 — minStock:alert doc id per-SKU 패턴

### 정정 대상

spec_v24 §3-9 "자동 발행 로그 (supplementMin:alert)"에 "일자 묶음 vs per-SKU 결정 필요"로 미확정 표기됨. 단위 11에서 per-SKU 패턴으로 확정.

### v25 확정

```
### 3-9. 자동 발행 로그 — per-SKU 패턴 확정 (v25 정정)

기존 봉투/원육 `minStock:alert` 패턴 그대로 통합. 봉투/원육과 같은 action/subAction 사용,
영양제 식별은 `details.kind` 필드로.

#### doc id 패턴

`auto_minStock_alert_{date}_{dedupKey}`

dedupKey 예시:
- 봉투: `bagType_${bagTypeId}`
- 원육: `meatType_${meatTypeId}`
- 영양제: `supplementStock_${supplementTypeId}`

같은 날 같은 SKU 1건만 존재. 같은 날 5봉 미만 → 5봉 이상 → 5봉 미만 → 5봉 이상...
변화해도 doc id는 동일이므로 1건으로 덮어쓰기.

#### 필드 구조

```js
{
  date: 'YYYY-MM-DD',
  timestamp: Timestamp,
  action: 'minStock',
  subAction: 'alert',
  details: {
    kind: 'supplement' | 'bag' | 'meat',  // 자재 종류 식별
    itemId: string,                       // supplementTypeId / bagTypeId / meatTypeId
    itemName: string,                     // 스냅샷
    currentQty: number,
    minimumQty: number,
  },
  isAutoGenerated: true,
  acknowledged: boolean,
  acknowledgedAt?: Timestamp,
  acknowledgedBy?: string,
}
```

#### 발행 시점

매일 1회. `triggerMinStockLogs(today)` 통합 호출에서 봉투/원육/영양제 모두 처리.
조회 시점 = 각 자재 메뉴 진입 또는 메인 진입 시 1회 (`ensureAutoLog()` 헬퍼).

#### 분류

사무 로그 (`OFFICE_LOG_ACTIONS`에 `minStock` 포함).
일반 생산/사무 로그 차단에는 미포함 (경고 전담, spec_v22 §4 유지).
```
