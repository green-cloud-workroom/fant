## 1. 배경 / 문제

현재 생산카드 저장 = `productions` doc + 영양제 차감만. **생산한 제품이 어떤 재고로도 들어가지 않음.** 생식 완제품 재고는 컬렉션 자체가 없고, 동결건조 빵판/동결판/동결제품은 수동 입력만(생산카드와 무링크). 판→팩 환산 데이터(고양이/강아지 판당 팩수 다름)도 데이터 모델에 없음.

목표: **생산카드에서 "제품 입고"로 실제 생산량(판/팩)을 입력 → 자동 환산 → 적절한 재고로 입고**. 최종 완제품은 재고앱으로 전송.

| 공용 레시피 판당팩수 | **확정: 설정에 cat/dog/공용 3개 모두 둠.** target 빈값 레시피는 공용값 사용 |

남은 미결 = **O6(완제품 재고앱 전송 메커니즘·스키마·수신책임) — 재고앱 협의** 뿐.

판당팩수 저장: `settings/productConversion = { cat, dog, common, updatedAt, updatedBy }`. 1박스=20팩 고정(O7, 코드 하드코딩 유지, 설정 미포함).

## 3. 데이터 흐름 (확정 결정 기반)

```
[생식 raw]
 생산카드 → 제품입고 모달(판/팩) → 환산(완제품 박스/팩) → 재고앱 전송 → 재고앱 완제품 입고

[동결건조 freezeDry, requiresSeparation=true (생식동결)]
 생산카드 → 제품입고 모달(판) → 빵판 입고(생산앱 breadPanLots)
   → [동결 분리작업 frozenSep] → 동결판(frozenPanLots)
   → 동결제품 입고(frozenProduct) → 재고앱 전송 → 재고앱 완제품 입고

[동결건조 freezeDry, requiresSeparation=false (텐더동결)]
 생산카드 → 제품입고 모달(판) → 동결판 직행(frozenPanLots, 빵판 skip)
   → 동결제품 입고(frozenProduct) → 재고앱 전송 → 재고앱 완제품 입고
```

→ 재고앱 전송 시점 = **완제품 단계**: 생식은 생산카드 제품입고 즉시(생식은 바로 완제품), 동결건조는 파이프라인 끝의 `frozenProduct` 입고 시점.

## 4. 데이터 모델 (생산앱 측)

### 4-1. 판당 팩수 환산 설정 — 신규 (D3)

설정 메뉴에 신규 섹션. 저장 위치 후보 = `settings/productConversion` (단일 doc) 또는 recipe별 필드.
- **권장(미확정)**: 레시피별 필드 — 각 레시피가 이미 대상(고양이/강아지)·카테고리를 가짐.
  - raw 레시피: `packsPerPlate` (1판 = N팩), `packsPerBox` (1박스 = N팩, 기본 20 — 현재 코드 하드코딩 `/20` 대체)
  - 단 "고양이/강아지 대상만으로 정해지고 레시피 무관"이면 설정 단일 doc에 `{ cat: {packsPerPlate}, dog: {packsPerPlate} }`.
- **미결 O1**: 판당팩수가 레시피별인지 대상(고양이/강아지)별인지 — 호두 확인. 설정 UI 형태 좌우.

### 4-2. 생식 완제품 — 재고앱 전송 (D4)

생식 완제품 재고는 **생산앱에 두지 않음**(재고앱 소유). 생산앱은 "완제품 입고 이벤트"를 재고앱이 읽을 수 있게 전송.
- 전송 메커니즘 후보(§6 협의): 공유 컬렉션 `productReceipts`(생산 write, 재고 read·처리) vs 재고앱 소유 컬렉션 직접 write.
- 페이로드(안): `{ source:'production', category:'raw', recipeId, recipeName, productionId, plates, packs, boxes, producedDate, staff, createdAt, app:'production' }`

### 4-3. 동결건조 빵판/동결판 자동입고 (D6)

기존 `breadPanLots`(빵판)·`frozenPanLots`(동결판) 컬렉션 재사용. 생산카드 제품입고가 자동 post:
- requiresSeparation=true → `breadPanLots`에 lot 생성(판수 기반) + `breadPanLogs`
- requiresSeparation=false → `frozenPanLots`에 직접 lot 생성 + `frozenPanLogs`
- **미결 O2**: 기존 수동 빵판/동결판 입고 모달은 유지(병행)인지, 생산카드 자동입고로 대체인지. (권장: 병행 — 수동은 예외 보정용)
- 자동 post 시 `relatedProductionId` 필드로 추적성 확보(영양제 패턴 차용).

### 4-4. 동결제품 → 재고앱 전송 (D5)

기존 `frozenProduct` 입고(frozenProduct.js)가 재고앱으로 전송 추가. 4-2와 동일 메커니즘, `category:'freezeDry'`.
- **미결 O3**: 동결제품 입고는 현재 수동. 이걸 그대로 두고 전송만 추가? 아니면 동결판→동결제품도 흐름 자동화?

## 5. UI 흐름 (생산앱 측)

### 5-1. 메인 생산카드 클릭 (D1)
- `main-production-card`에 클릭 핸들러 추가(현재 없음). 클릭 → 제품입고 모달.
- **미결 O4**: 메인 카드 클릭 = 제품입고 전용인지, 다른 액션(상세보기 등)과 공존인지.

### 5-2. 제품입고 모달 (D1, D2)
- 제목: `제품 입고 — {레시피명}`
- 생식: 판수 입력 → 자동 팩/박스 표시(설정 판당팩수·packsPerBox). "몇팩" 추가 입력(나머지 자투리) 허용 여부 = **미결 O5**.
- 동결건조: 판수 입력(빵판수 or 동결판수, requiresSeparation 따라). 봉지수 등 표시.
- 버튼: `제품 입고`(확정) / `취소`
- 확정 시: 위 4절 라우팅대로 재고 post + (완제품이면) 재고앱 전송.

## 6. Cross-app 계약 — 신규 역방향 (생산 → 재고)

⚠ 기존 Cross-app 요청서 v2(재고→생산)엔 **없던 신규 흐름.** 재고앱이 "생산앱發 완제품 입고"를 수신·처리하는 측을 만들어야 함. 양 앱 협의 필수.

협의 항목:
1. **전송 메커니즘**: 공유 컬렉션(`productReceipts` 등, 생산 write·재고 read) vs 재고앱 소유 컬렉션 직접 write vs Cloud Function. (재고앱 = asia-northeast3 Functions 보유; 생산앱 = Functions 없음)
2. **페이로드 스키마 합의**: §4-2 안 기준 확정.
3. **수신 책임**: 재고앱이 이 이벤트를 완제품 재고로 어떻게 반영하는지(자동 입고 vs 검수 후 입고).
4. **중복/멱등**: 같은 productionId 재전송 시 중복 입고 방지(idempotency key).
5. **firestore.rules**: 공유 컬렉션이면 생산 write·재고 read 규칙(생산앱 소유 rules에 추가).
6. **재고앱 Sprint 위치**: 재고앱 완제품 수신 기능이 어느 Sprint인지(현재 재고 Sprint1 = 마스터/주문 중심, 완제품 수신은 미정).

→ 이 6항목을 재고앱 측에 **신규 Cross-app 요청서(생산→재고)**로 전달 필요. (생산앱 `docs/fantapet_crossapp_response_2026-05-21.md`와 별개 신규 문서)

## 8. 구현 단계 (제안, 협의 후 확정)

1. **(생산앱) 판당팩수 설정 + recipe/settings 필드** — O1 확정 후
2. **(생산앱) 메인 생산카드 클릭 + 제품입고 모달 + 환산** — O4·O5 확정 후
3. **(생산앱) 동결건조 빵판/동결판 자동입고 (requiresSeparation 분기)** — O2 확정 후
4. **(cross-app) 완제품 전송 메커니즘 + 스키마** — O6 재고 협의 후 (양 repo 동시)
5. **(생산앱) 생식 완제품 + 동결제품 → 재고앱 전송** — 4 완료 후
6. **(재고앱) 완제품 수신·입고 처리** — 재고 Sprint

각 단계 dry-run/검증/호두 GO 게이트. cross-app 단계(4~6)는 양 앱 동시 + 멱등/롤백 설계.
