## 1. 결정 4 정정 (운영 시작 시점)

### v22 (정정 대상)
```
### 결정 4 — 진행 시점: A (운영 시작 후) — v22 재확인
- 생산관리 앱 운영 시작 후 별도 묶음으로 진행
- 재고관리 앱 개발은 그 이후
- 운영 1~2주 돌리며 발견되는 필드 변경 자연스럽게 반영 후 안정화된 마스터 위에서 개발
- v22 보강: 묶음 e 중 "운영 시작 전 진행" 권장 외부 문서가 작성되었으나 본 결정과 모순되어 폐기됨. 결정 4 유지 확정.
```

### v23 정정 (사용자 결정 반영)

**용어 정의**

| 용어 | 정의 |
|---|---|
| **단독 검증 완료** | 생산관리 앱이 단독으로 모든 기능 검증을 끝낸 상태. 이게 v19/v22에서 "운영 시작 진입 가능"으로 잘못 표기됐던 시점. |
| **동시 운영 시작** | 생산관리 앱 + 재고관리 앱이 모두 완료되어 둘 다 실데이터로 가동되는 시점. **이게 사용자 정의의 "운영 시작"**. |
| **시드 입력** | 단독 검증 완료 후, 재고관리 앱 묶음 진입 직전 또는 묶음 중에 입력하는 **테스트/마스터 데이터**. 실운영 데이터 아님. |

**결정 4 재해석**

```
### 결정 4 — 진행 시점: 재고관리 앱 묶음 진입 단계로 재해석 (v23 정정)

원문 "운영 시작 후"는 "단독 검증 완료 후"로 정정.
즉 현재 상태 = 재고관리 앱 묶음 진입 가능.

- 생산관리 앱 단독 검증 완료 후 재고관리 앱 묶음 진입 (현재 단계)
- 재고관리 앱 묶음 진행 중 시드 입력으로 마스터 데이터 확보
- 운영 1~2주 안정화는 "동시 운영 시작 후"로 의미 변경
  - 재고관리 앱 묶음 완료 + 동시 운영 시작 후 1~2주 = spec_v24 작성 시점

- v22 보강 (폐기 문서 부분 복원):
  - 폐기 사유였던 "운영 시작 전 진행" 부분이 본 정정에 맞게 재해석되어 일부 항목 복원됨
  - 휴일 마스터 항목만 Phase 2로 편입 (이 spec 20절 결정 8에 신설)
  - 나머지 재고관리 앱 추가 작업 항목은 재고관리 앱 묶음 spec에서 다룸
```

## 2. 7절 — 텐더동결 빵판 표시 정책 (B안 반영)

### 신설 항목 (7절 "동결건조 레시피 추가 설정" 내)

```
### 동결건조 레시피 추가 설정 (v23 정정)

- 생산 단위 1당 제품 봉지수
- 생산 단위 1당 빵판 수
  - **분리 작업 필요 = false (텐더동결) 인 경우 UI에서 input 숨김**
  - **저장 시 breadPanCountPerUnit: 0 강제**
- 생산 단위 1당 동결판 수
- 분리 작업 필요 여부 체크박스
  - 변경 시 빵판수 input 즉시 show/hide 반응
```

### 신설 항목 (8절 "생산 카드" 내)

```
- 카드 내용:
  - 생식: 박스수 표시
  - 동결건조 + 분리작업 필요 (생식동결): 봉지수·빵판수·동결판수 표시
  - **동결건조 + 분리작업 불필요 (텐더동결): 봉지수·동결판수만 표시 (빵판수 줄 생략)**
```

### 신설 항목 (8절 "생산지시서" 내)

```
- 텐더동결 섹션에서 빵판 줄 출력 생략
- 동결판수/봉지수만 표시
```

### 데이터 구조 (참고)

```
recipes 컬렉션:
- requiresSeparation: false → 텐더동결, 빵판 흐름 무관
- requiresSeparation: true  → 생식동결, 빵판 흐름 포함

UI는 requiresSeparation 값으로 빵판 관련 표시 분기.
DB 구조는 v22와 동일 (category=freezeDry + requiresSeparation 토글).
```

## 3. 9·11·12절 — 마스터 토글 위치 정책 통일 (C-1 반영)

### 정책 (v23 신설)

```
### 마스터 활성/비활성 토글 위치 — 통일 패턴

대상 마스터: 원육 / 봉투 / 동결제품 / 레시피

통일 패턴: **목록 행에 직접 토글 배치**
- 수정 모달 안에 토글 두지 않음
- 등록 후 행에서 즉시 ON/OFF
- production 계정에서는 토글 DOM 제거 유지
- 토글 변경 시 activityLogs에 activeToggle 기록

이유:
- 직원이 마스터 메뉴별로 다른 토글 위치 외울 필요 없음
- 클릭 1회로 변경 가능
- 활성/비활성 상태 한눈에 확인 가능

⚠ 재고관리 앱 묶음에서 신규 마스터 도입 시 동일 패턴 적용 필수.
```

### 9절 원육 (변경 없음)
v22 그대로 (기준 패턴이었음)

### 11절 봉투 (v23 정정)

- 봉투 추가 모달: 봉투명 / 종류 / 박스당 장수 / 최소재고 필드만
- **활성/비활성 토글은 수정 모달 아닌 목록 행에 배치** (v23)
- 비활성 봉투는 신규 입력 드롭다운에서 제외 (v22 유지)

## 4. 20절 결정 8 — 휴일 마스터 항목 추가

### 신설 (재고관리 앱 연동 — 휴일/특수일 마스터)

폐기 문서(2026-05-11 작성)에서 휴일 마스터 부분만 복원. 나머지 재고관리 앱 추가 작업은 재고관리 앱 묶음 spec에서 다룸.

```
### 결정 9 — 휴일 마스터 (v23 신설)

생산관리 앱이 휴일/특수일 마스터 보유. 재고관리 앱은 참조만.

#### A. 한국 공휴일 자동 등록
- 향후 5년치 공휴일 코드에 내장 (A안)
- 매년 1회 데이터 업데이트만 하면 됨
- 외부 API 미사용 (의존성 회피)

#### 자동 등록 대상
- 신정 (1/1)
- 설날 + 전후 1일 (음력 1/1)
- 삼일절 (3/1)
- 어린이날 (5/5)
- 부처님오신날 (음력 4/8)
- 현충일 (6/6)
- 광복절 (8/15)
- 추석 + 전후 1일 (음력 8/15)
- 개천절 (10/3)
- 한글날 (10/9)
- 성탄절 (12/25)
- 대체공휴일 (정부 발표 기준)

#### 자동 등록 시 기본값
- holidayType = 'publicHoliday'
- affectsProduction = true
- affectsShipping = true
- shippingClosedFromEnabled = true
- isAutoGenerated = true

#### B. Holiday 컬렉션 필드 (v22 대비 확장)
{
  holidayId: string,
  date: date,
  holidayType: 'publicHoliday' | 'internalOff' | 'parcelClose' | 'parcelResume' | 'productionOff' | 'shippingOff' | 'traditional' | 'recurring',
  title: string,
  description: string,
  affectsProduction: boolean,
  affectsShipping: boolean,
  shippingClosedFromEnabled: boolean,
  isAutoGenerated: boolean,
  recurrenceRule: json | null,
  createdBy: string,
  updatedAt: timestamp
}

#### C. 배송 불가일 자동 확장 룰
- 공휴일/대체공휴일/명절/택배 마감일: shippingClosedFromEnabled=true (전날 배송 불가)
- 내부 휴무일/생산 불가일: shippingClosedFromEnabled=false (당일만 불가)
- 운영자 수동 변경 가능

#### D. 토/일 처리
- 토/일은 명시적 Holiday 레코드 없음
- 시스템 설정값으로 처리

#### E. 운영자 변경 가능 사항
- affectsProduction / affectsShipping / shippingClosedFromEnabled
- 변경 시 이력 저장 (변경자/일시/전·후 값)

#### F. 진행 시점
- Phase 2 (재고관리 앱 묶음 진입 직전 필수)
- 묶음 9 #22 (캘린더 휴일 표시)와 같이 작업
- 생산관리 앱은 affectsProduction만 보면 됨

#### G. 운영자 시드 입력 시
- 한국 법정공휴일: 입력하지 않음 (자동 등록 대상)
- 회사 휴무일: 입력 가능 (마이그레이션 시 디폴트값 자동 채워짐)
```
